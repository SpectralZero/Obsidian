## 1. What is Detection Engineering?

### 1.1 The Simple Definition

**Detection Engineering** = The process of designing, building, testing, and maintaining rules that identify malicious activity in your environment.

Think of it like **programming an alarm system**:
- **Bad alarm**: Goes off every time a cat walks by (false positives)
- **Good alarm**: Only alerts when an actual intruder is detected
- **Detection Engineering**: Creating the "good alarm" for cyber attacks

### 1.2 Detection Engineer vs SOC Analyst

```
┌────────────────────────────────────────────────┐
│         ROLE COMPARISON                        │
├────────────────────────────────────────────────┤
│ SOC ANALYST (Tier 1/2)                         │
│ • Uses existing detection rules                │
│ • Investigates alerts generated by rules       │
│ • Reactive: Responds to what's detected        │
│ • Salary: $45k-$95k                            │
├────────────────────────────────────────────────┤
│ DETECTION ENGINEER (Senior)                    │
│ • Creates new detection rules                  │
│ • Improves existing rules                      │
│ • Proactive: Hunts for coverage gaps           │
│ • Salary: $95k-$150k+                          │
└────────────────────────────────────────────────┘
```

**Career Path**: SOC Analyst → Senior Analyst → Detection Engineer → Detection Lead

### 1.3 Why Detection Engineering Matters

**The Problem with Default Rules**:
```
OUT-OF-BOX SIEM RULES:
❌ Generic (not tailored to YOUR environment)
❌ High false positive rate (80%+)
❌ Miss custom attacks
❌ Outdated (don't catch new techniques)

CUSTOM DETECTION ENGINEERING:
✅ Tailored to your environment
✅ Low false positives (<10%)
✅ Catches sophisticated attacks
✅ Continuously improved
```

**Real-World Impact**:
- Organizations with mature detection engineering: **50% faster breach detection**
- Custom detections catch **60% more threats** than default rules
- Reduced false positives = **3x more efficient SOC**

## 2. The Detection Development Lifecycle

### 2.1 The 7-Step Process

```
┌────────────────────────────────────────────────┐
│      DETECTION DEVELOPMENT LIFECYCLE           │
└────────────────────────────────────────────────┘

STEP 1: THREAT RESEARCH
├─ What attack are we detecting?
├─ How does the attack work?
├─ What logs does it generate?
└─ Example: Research Kerberoasting attack

STEP 2: DATA SOURCE IDENTIFICATION
├─ What logs do we need?
├─ Do we have those logs?
├─ Are they going to SIEM?
└─ Example: Need Windows Event ID 4769

STEP 3: DETECTION LOGIC DESIGN
├─ What pattern indicates attack?
├─ Write pseudocode logic
├─ Define thresholds
└─ Example: >10 service tickets in 1 hour

STEP 4: RULE DEVELOPMENT
├─ Write rule in SIEM query language
├─ Add metadata (severity, description)
├─ Define response actions
└─ Example: SPL/KQL/Sigma rule

STEP 5: TESTING
├─ Test on historical data
├─ Simulate attack in lab
├─ Measure false positive rate
└─ Example: Run against 30 days of logs

STEP 6: DEPLOYMENT
├─ Deploy to production SIEM
├─ Set appropriate schedule
├─ Configure alerting
└─ Example: Run every 15 minutes

STEP 7: CONTINUOUS IMPROVEMENT
├─ Monitor performance
├─ Tune based on feedback
├─ Update as attacks evolve
└─ Example: Reduce FP threshold from 10→15
```

### 2.2 Detection Rule Anatomy

**Every Detection Rule Has**:

```yaml
RULE METADATA:
  - Name: Clear, descriptive name
  - ID: Unique identifier
  - Author: Who created it
  - Date: When created
  - Version: For tracking changes
  - References: ATT&CK, blog posts, CVEs

DETECTION LOGIC:
  - Data Source: What logs to query
  - Conditions: What patterns to match
  - Thresholds: Numeric limits (>5, <100, etc.)
  - Time Window: How long to look back

RESPONSE:
  - Severity: Critical, High, Medium, Low
  - Alert Title: What analyst sees
  - Description: What happened
  - Recommended Actions: What to do
  - False Positive Notes: Known FPs
```

## 3. Understanding Detection Logic & Pattern Recognition

### 3.1 Types of Detection Logic

**1. Signature-Based (Exact Match)**
```
LOGIC: IF event matches exact pattern THEN alert

EXAMPLE:
IF filename = "mimikatz.exe" THEN alert

PROS:
✅ High accuracy (low FP)
✅ Fast detection
✅ Easy to understand

CONS:
❌ Easy to bypass (rename file)
❌ Doesn't catch variants
```

**2. Anomaly-Based (Statistical)**
```
LOGIC: IF behavior deviates from baseline THEN alert

EXAMPLE:
IF login_count > avg_login_count * 3 THEN alert

PROS:
✅ Catches unknown attacks
✅ Adapts to environment

CONS:
❌ Higher false positives
❌ Requires baseline period
```

**3. Behavior-Based (Heuristic)**
```
LOGIC: IF sequence of actions matches attack pattern THEN alert

EXAMPLE:
IF (port_scan AND ssh_login AND privilege_escalation) THEN alert

PROS:
✅ Catches multi-stage attacks
✅ Hard to evade

CONS:
❌ Complex to build
❌ Requires correlation
```

**4. Threshold-Based**
```
LOGIC: IF count exceeds threshold THEN alert

EXAMPLE:
IF failed_logins > 10 in 5 minutes THEN alert

PROS:
✅ Simple to implement
✅ Low FP if tuned right

CONS:
❌ Slow attacks evade (9 attempts = no alert)
```

### 3.2 Building Detection Logic (Pseudocode First!)

**ALWAYS start with pseudocode before writing SIEM queries**

**Example: Detect Brute Force**

```
PSEUDOCODE:
1. Get all failed login events
2. Group by (user, source_ip)
3. Count failures per 5-minute window
4. If count > 10, generate alert
5. Include: user, IP, count, time

SPLUNK SPL:
index=windows EventCode=4625
| bucket _time span=5m
| stats count by _time, user, src_ip
| where count > 10

ELASTIC KQL:
event.code: "4625"
| stats count by user.name, source.ip
| where count > 10

SENTINEL KQL:
SecurityEvent
| where EventID == 4625
| summarize count() by bin(TimeGenerated, 5m), Account, IpAddress
| where count_ > 10
```

## 4. Writing Your First Detection Rule

### 4.1 Scenario: Detect Suspicious PowerShell

**STEP 1: Research the Threat**

PowerShell is heavily used by attackers for:
- Downloading malware
- Running encoded commands
- Bypassing execution policies
- Living-off-the-land attacks

**Suspicious indicators**:
- `-EncodedCommand` (base64 encoded payload)
- `-WindowStyle Hidden` (hide window)
- `-ExecutionPolicy Bypass` (bypass security)
- `-NoProfile` (don't load profile)

**STEP 2: Identify Data Source**

Windows Event ID 4688 (Process Creation) with command line logging enabled

**STEP 3: Write Pseudocode**

```
IF process = PowerShell.exe
AND command_line contains (-enc OR -noprofile OR -windowstyle hidden OR -ep bypass)
THEN alert
```

**STEP 4: Write SIEM Rule**

**Splunk SPL**:
```spl
index=windows EventCode=4688 NewProcessName="*powershell.exe"
| rex field=CommandLine "(?i)(-enc|-noprofile|-windowstyle\s+hidden|-ep\s+bypass)"
| where isnotnull(CommandLine)
| table _time, Computer, User, CommandLine
| eval severity="HIGH"
| eval description="Suspicious PowerShell execution detected"
| eval recommendation="1. Isolate endpoint 2. Check for malware 3. Review recent user activity"
```

**STEP 5: Test the Rule**

```spl
# Simulate attack
C:\> powershell.exe -enc SGVsbG8gV29ybGQh

# Check if rule triggers
index=windows EventCode=4688 NewProcessName="*powershell.exe" earliest=-5m
```

**STEP 6: Deploy & Monitor**

```
Save As: Alert
Name: Suspicious PowerShell Execution
Schedule: Every 5 minutes
Trigger: Number of Results > 0
Action: Email SOC team
```

## 5. MITRE ATT&CK Framework for Detection

### 5.1 Understanding ATT&CK

**ATT&CK** = Adversarial Tactics, Techniques, and Common Knowledge

A framework of 14 tactics and 185+ techniques used by attackers.

**The 14 Tactics** (the "why"):
```
1. Reconnaissance       → Gather info about target
2. Resource Development → Build infrastructure
3. Initial Access       → Get into network
4. Execution            → Run malicious code
5. Persistence          → Maintain foothold
6. Privilege Escalation → Get higher privileges
7. Defense Evasion      → Avoid detection
8. Credential Access    → Steal passwords
9. Discovery            → Explore environment
10. Lateral Movement    → Spread across network
11. Collection          → Gather data
12. Command & Control   → Communicate with attacker
13. Exfiltration        → Steal data out
14. Impact              → Destroy/encrypt (ransomware)
```

### 5.2 Mapping Detections to ATT&CK

**Example: Kerberoasting Detection**

```
ATT&CK MAPPING:
├─ Tactic: Credential Access (TA0006)
├─ Technique: Kerberoasting (T1558.003)
├─ Data Source: Windows Event Logs
└─ Event: 4769 (Kerberos Service Ticket Request)

DETECTION LOGIC:
IF Event_ID = 4769
AND Ticket_Encryption_Type = 0x17 (RC4)
AND Service_Account_Count > 10
AND Time_Window = 1 hour
THEN Alert "Kerberoasting Detected"
```

**Coverage Matrix**:
```
Your Goal: Cover as many ATT&CK techniques as possible

Current Coverage Example:
├─ Initial Access: 5/9 techniques (56%)
├─ Execution: 8/13 techniques (62%)
├─ Persistence: 12/19 techniques (63%)
├─ Privilege Escalation: 10/14 techniques (71%)
└─ Overall: 89/185 techniques (48%)

Use ATT&CK Navigator to track coverage!
```

## 6. Sigma Rules - Universal Detection Format

### 6.1 What is Sigma?

**Sigma** = A generic signature format for SIEM rules (like YARA for logs)

**Why Sigma?**
- ✅ Write once, run anywhere (Splunk, Elastic, Sentinel, etc.)
- ✅ Shareable across organizations
- ✅ Version controlled (YAML format)
- ✅ Large community library (2000+ rules)

### 6.2 Sigma Rule Structure

```yaml
title: Suspicious PowerShell Execution
id: a3d77fbd-7123-4c09-8b05-5c5c45ad89db
status: stable
description: Detects suspicious PowerShell command line parameters
author: Your Name
date: 2024/01/31
references:
  - https://attack.mitre.org/techniques/T1059/001/
tags:
  - attack.execution
  - attack.t1059.001
logsource:
  product: windows
  service: security
  definition: 'Requirements: Audit Process Creation, Command Line logging'
detection:
  selection:
    EventID: 4688
    NewProcessName|endswith: '\powershell.exe'
    CommandLine|contains:
      - '-enc'
      - '-EncodedCommand'
      - '-w hidden'
      - '-WindowStyle hidden'
      - '-nop'
      - '-noprofile'
      - '-ep bypass'
      - '-ExecutionPolicy bypass'
  condition: selection
falsepositives:
  - Administrative scripts (rare)
  - Legitimate automation tools
level: high
```

### 6.3 Converting Sigma to SIEM-Specific Rules

**Install Sigma Tools**:
```bash
pip install sigma-cli

# Convert to Splunk
sigma convert -t splunk rule.yml

# Convert to Elastic
sigma convert -t elasticsearch rule.yml

# Convert to Sentinel
sigma convert -t azure rule.yml
```

**Output (Splunk)**:
```spl
EventCode=4688 NewProcessName="*\\powershell.exe" 
(CommandLine="*-enc*" OR CommandLine="*-EncodedCommand*" 
 OR CommandLine="*-w hidden*" OR CommandLine="*-WindowStyle hidden*")
```

## 7. Platform-Specific Rule Writing

### 7.1 Splunk Detection Rules

```spl
# Saved Search / Alert
index=windows EventCode=4625
| bucket _time span=5m
| stats count by _time, user, src_ip
| where count > 10
| eval severity="HIGH"
| eval attack_type="Brute Force"
| eval recommendation="Block IP, reset password, investigate account"
| table _time, user, src_ip, count, severity, attack_type, recommendation
```

**Save as Alert**:
- Name: Brute Force Detection
- Schedule: `*/5 * * * *` (every 5 min)
- Trigger: Results > 0
- Actions: Email + Webhook

### 7.2 Elastic Detection Rules

**Create Rule in Kibana**:
```
Security → Rules → Create New Rule

Rule Type: Custom Query
Index Pattern: winlogbeat-*

Query:
event.code: "4625"
AND agent.type: "winlogbeat"

Aggregation:
count() by user.name, source.ip

Condition:
count() > 10 within 5 minutes

Actions:
- Send email
- Create case
- Run webhook
```

### 7.3 Sentinel Analytics Rules

```kusto
SecurityEvent
| where EventID == 4625
| summarize count() by bin(TimeGenerated, 5m), Account, IpAddress
| where count_ > 10
| extend Severity = "High"
| extend ThreatType = "Brute Force"
| extend Recommendation = "Block IP, reset password"
| project TimeGenerated, Account, IpAddress, count_, Severity, ThreatType
```

**Save as Analytics Rule**:
```
Rule Type: Scheduled
Run every: 5 minutes
Lookup data from last: 10 minutes
Alert threshold: Greater than 0
Entity mapping:
  - Account → Account
  - IP Address → IP
Incident: Create incident
Severity: High
```
